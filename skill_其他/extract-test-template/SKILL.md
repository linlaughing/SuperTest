# 测试用例格式模板提取器

分析现有测试用例代码，提取结构化的测试格式模板文档。

## 输入

用户提供以下之一：
- 一个或多个**测试用例文件路径**（`.py` 或 `.json`）
- 一个**测试目录路径**（如 `cases/agent/react_agent/`）
- 一段**模块描述**（如 "ReactAgent 测试"）

## 输出

保存到 `ai_reference/` 目录的 markdown 模板文档，格式参照 `ai_reference/test_common_template.md`。

## 工作流程

### 1. 确定分析范围

- **文件路径输入**：直接读取指定文件
- **目录输入**：扫描目录下所有 `.py` 和 `.json` 测试文件，按文件名排序后全部读取
- **模块描述输入**：在 `cases/` 下查找匹配的目录，然后同上处理

注意：用例可能是单独的 `.py` 文件，也可能是 `.json` 文件描述的用例。两种格式都要支持。

### 2. 读取框架参考（可选上下文）

如果 `ai_reference/framework_reference.md` 存在，读取其中与目标类别相关的 fixture 和工具类信息，用于在模板中正确引用 fixture 名称和辅助类路径。

### 3. 逐文件提取模式

对每个测试文件，提取以下维度：

**3.1 文件头部**
- 编码声明（如 `# -*- coding: utf-8 -*-`）
- Copyright 声明
- 标准库导入
- 框架导入（`from openjiuwen.*`）
- 测试工具导入（`from common.*`）

**3.2 函数签名**
- 装饰器列表及顺序（`@pytest.mark.*`）
- fixture 参数列表
- 函数命名格式

**3.3 文档字符串**
- 是否有 docstring
- docstring 中的字段列表（@CaseID, @Description, @Step, @Result 等）
- 各字段的值格式和写法风格

**3.4 测试主体**
- `test_resource_manager` 的使用方式（参数传递、资源获取）
- 测试对象的创建和配置步骤
- 执行调用方式（invoke / stream / run_agent / run_agent_streaming）
- 关键位置的 `logger.info` 使用习惯

**3.5 断言模式**
- assert 语句的目标字段（如 `state.value`, `result_type`, `len()`）
- 正向/逆向断言风格
- 错误场景断言方式（异常捕获、错误码检查）

**3.6 JSON 用例模式**（如果存在 .json 文件）
- JSON 的 schema 结构
- 字段含义和取值范围
- 与 .py 文件的关联关系

### 4. 模式归并

将所有文件中提取的模式按出现频率排序：
- **导入路径**：统计每个 import 的出现次数，只取高频导入
- **装饰器组合**：统计常见的装饰器组合，只取主流组合
- **fixture 参数**：统计 fixture 参数的出现频率，只取高频 fixture
- **执行方式**：统计各执行方式的出现次数，只取主要方式
- **断言字段**：统计常见的断言目标，只取高频断言模式

**只保留高频通用模式**，不生成场景特定补充章节，避免模板膨胀影响后续自动化用例生成。
