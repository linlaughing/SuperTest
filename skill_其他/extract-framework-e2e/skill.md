# E2E 框架场景分析器

分析任意代码框架的端到端测试场景，通过自动探测框架结构识别模块、类、依赖关系，生成框架全局 E2E 场景文件。

## 参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `framework_root` | 框架根目录（必填） | - |
| `--output` | 输出文件路径 | `ai_reference/e2e_framework_scenes.md` |
| `--output-dir` | 输出目录 | `ai_reference/` |

### 示例

```bash
/extract-framework-e2e /path/to/framework
/extract-framework-e2e /path/to/framework --output-dir output/
```

## 执行步骤

### 第一步：全局架构探测

通过代码结构分析自动识别框架的模块组织和功能定位。

1. **识别目录与模块结构**
   - 使用 Glob 遍历 `framework_root` 下所有 Python 包（含 `__init__.py` 的目录）
   - 记录每个包的层级关系（顶层模块 → 子模块）
   - 识别各子模块的规模（文件数、类数）以判断功能定位

2. **提取核心类与接口**
   - 对每个模块使用 Grep 搜索 `class ` 定义，记录类名和继承关系
   - 对每个核心类使用 Grep 搜索 `def ` 方法，识别公开 API
   - 识别类的初始化参数（`__init__` 方法签名）

3. **分析模块间依赖关系**
   - 在每个模块内 Grep 搜索 `from {framework_root}` 或 `import {framework_root}` 的导入语句
   - 根据导入方向构建模块依赖图（被导入 = 基础组件，导入他人 = 上层业务）
   - 识别跨模块的类继承关系

4. **识别框架功能分类**

   基于探测结果，将模块自动归类到功能分类中。使用以下启发式规则：

   | 功能分类 | 识别信号 |
   |----------|----------|
   | **核心引擎** | 模块被大量其他模块导入；包含 `Engine`、`Runner`、`Manager`、`Controller` 类 |
   | **数据/存储** | 类名含 `Store`、`Database`、`Repository`、`Cache`、`Storage`；方法含 `save`、`load`、`query`、`delete` |
   | **通信/网络** | 类名含 `Client`、`Server`、`Handler`、`Adapter`、`Transport`；方法含 `request`、`send`、`connect` |
   | **编排/流程** | 类名含 `Workflow`、`Pipeline`、`Flow`、`Graph`、`DAG`；方法含 `run`、`execute`、`start`、`next` |
   | **插件/扩展** | 类名含 `Plugin`、`Extension`、`Hook`、`Middleware`、`Tool`；被核心模块动态加载 |
   | **配置/模型** | 类名含 `Config`、`Setting`、`Option`、`Model`、`Schema`；以数据属性为主，方法少 |
   | **工具/辅助** | 类名含 `Util`、`Helper`、`Builder`、`Factory`；无状态或纯函数 |

   > 以上分类仅为启发式参考。如果模块不匹配任何分类，根据其实际功能自定义分类名称。分类数量不限定，以清晰描述框架结构为准。

### 第二步：端到端场景识别

基于第一步的架构探测结果，识别所有端到端测试场景。

#### 场景生成方法

不预设场景分类维度，而是基于探测到的模块和功能自动生成场景：

1. **单模块场景**：每个具备独立业务入口的模块（有公开的 `run`/`execute`/`invoke` 等入口方法）
   - 场景命名：`{模块功能}_{核心操作}`
   - 涉及模块：该模块及其直接依赖
   - 入口点：模块的核心入口方法

2. **跨模块协作场景**：模块间存在调用关系的组合
   - 从依赖图中找到直接调用关系（A 模块 import B 模块并调用其 API）
   - 场景命名：`{调用方功能}_with_{被调用方功能}`
   - 涉及模块：调用双方
   - 入口点：调用方的入口方法

3. **完整业务流程场景**：涉及3个及以上模块的长链路
   - 从依赖图中找到深度 >= 2 的调用链（A → B → C）
   - 场景命名：描述完整业务目标的名称
   - 涉及模块：调用链上所有模块
   - 入口点：链路起点模块的入口方法

#### 场景数量控制

| 模块类型 | 场景数量 |
|----------|----------|
| 核心引擎 | 每个 `run`/`execute` 类入口方法 1-3 个场景 |
| 编排/流程 | 每种编排模式（线性、分支、循环、并行等）1-2 个场景 |
| 数据/存储 | 每种存储后端 1 个场景 + 跨存储协作 1 个场景 |
| 通信/网络 | 每种协议/传输方式 1 个场景 |
| 插件/扩展 | 每种插件类型 1 个场景 + 多插件组合 1 个场景 |
| 跨模块协作 | 每对有调用关系的模块 1-2 个场景 |
| 完整业务流程 | 每条深度 >= 2 的调用链 1 个场景 |

#### 场景排除规则

- 排除单接口测试、单组件测试、单模块测试（只验证一个方法返回值的场景）
- 排除纯数据模型/配置类场景（无业务流程的场景）
- 排除没有端到端流程的单独功能场景（如只构造对象不执行业务）
- 保留所有涉及多组件组合的完整业务流程场景

#### 场景 ID 命名规则

```
E2E_{分类标识}_{序号}
```

- 分类标识：根据第一步的功能分类自动生成（如 `CORE`、`STORE`、`FLOW`、`COMBO` 等）
- 序号：同一分类内从 001 递增
- 命名应让使用者直观理解场景含义
