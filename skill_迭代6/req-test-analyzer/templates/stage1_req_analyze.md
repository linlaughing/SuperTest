## 执行步骤（严格按顺序）

### 第一步：读取参考文件（并行 Read）

必须读取以下文件：

### 第二步：读取输入

Read `{requirement_doc}`（需求文档）

如果需求文档中引用了子文档，先读取每个子文档的**标题+首段描述**进行分层：

| 层级 | 判定条件 | 处理 |
|------|---------|------|
| 用户层 | 描述"对外接口"/用户可直接使用的功能/API | 全量读取 |
| 内部层 | 描述"子模块"/"内部"/"工具"/"负责XX处理"/"供XX调用" | 标记为`internal`，仅用于已有FP的exception/constraint补充 |

**判定方法**：主文档对子文档的引用说明（如"extraction: 实体与关系抽取"→内部层；"graph_memory: 对外接口"→用户层）。不确定时默认用户层。

分层结果记录到`doc_layers`变量，供后续步骤引用。

### 第三步：文档解构 + 主流程识别

#### 3a. 7维度信息源识别

| 维度 | 提取内容 | 典型来源 |
|------|---------|---------|
| 功能点 | 交互步骤、分支路径 | 流程描述、数据流 |
| 接口契约 | 方法签名、参数、返回值 | 接口定义、SDK说明 |
| 数据模型 | 数据结构、字段约束 | 存储结构、数据表 |
| 状态转换 | 正常/异常状态流转 | 时序图、降级方案 |
| 质量属性 | 性能/安全/可靠性指标 | 约束章节、非功能需求 |
| 业务规则 | 计算规则、校验逻辑 | 查询转换、处理策略 |
| 边界异常 | 错误场景、降级路径 | 失败条件、参数校验 |

**文档类型自适应**：有方法签名→侧重接口契约；有时序图→侧重数据流+状态转换；有用户故事→侧重主流程+业务规则；混合型→全维度。

#### 3b. 代码示例提取

提取文档中展示**用户入口调用**的代码示例 → `{output_dir}/.state/skeleton/`。无代码示例则不创建。

**过滤规则**：仅提取展示用户可独立触发入口的示例（引用第四步FP单入口原则）。跳过展示内部方法（带`_`前缀、processor/handler等内部组件）的示例。

如 `{work_dir}/ai_reference/test_req_template.md` 存在，拷贝到 `{output_dir}/.state/skeleton/req_template.md`。

#### 3c. 主流程链路识别

**代码示例优先**：如文档包含可执行代码示例（如"快速开始"/"Quick Start"），该示例展示的完整调用链即为**第一主流程**，直接作为FS-001。其他主流程从文档的独立章节（如"恢复与持久化"、"健康检查"等非快速开始的章节）提取。

**链路完整性要求**：一条主流程 = 用户从发起操作到获得最终结果的**完整端到端路径**。禁止将一条完整链路拆为"前半段"和"后半段"两个主流程。

对每条识别到的主流程，提取四要素：

| 要素 | 说明 | 识别方法 |
|------|------|---------|
| 关键步骤 | 端到端操作序列（必须包含完整生命周期：初始化→操作→验证→收尾） | 从编号列表/流程图/代码示例提取 |
| error_paths | 每个步骤的失败点 | 三层追问法：（1）显式失败：文档明确描述的失败场景；（2）隐含条件：该步骤暗含什么前提？前提不满足时怎样？（如"点击登录"隐含"已输入凭证"）；（3）隐含异常：需求未说明但合理可推断的异常（如超时、资源耗尽、并发冲突） |
| variants | 配置不同导致的变体路径 | 检查配置项是否影响行为+不同取值是否导致不同路径 |
| data_scope | 每个步骤处理的数据实体 | 对每步追问：该步骤读写了哪些数据？ |

### 第四步：功能点提取

从第三步提取原子功能点，映射为 FP。

**来源优先级**（所有来源必须通过三原则过滤）：

| 来源 | 说明 |
|------|------|
| 主流程步骤 | 每个关键步骤→1个FP |
| 接口方法 | 每个方法→候选FP，再按合并/拆分规则调整 |
| 配置项 | 仅当用户可独立触发该配置时创建FP，否则归入已有FP的constraints |

**FP 粒度判定规则（三原则）**：

| 原则 | 规则 | 示例 |
|------|------|------|
| 单入口 | 1个FP = 1个用户可独立触发的入口 | ✅ "创建Agent" = 1个FP；❌ "创建Agent+配置工具" = 2个FP |
| 单行为 | 1个FP = 1个原子用户行为（主语+谓语+宾语） | ✅ "配置重试策略" = 1个FP；❌ "配置重试和超时" = 2个FP |
| 可验证 | 1个FP必须能独立验证成功/失败 | ✅ "流式输出" = 1个FP；❌ "执行过程" = 不可独立验证 |

**合并规则**（满足全部条件时合并为1个FP）：
- 多个方法属于同一用户操作链路的连续步骤（如"加载YAML配置→验证→构建Leader"属于"构建团队实例"）
- 它们共享相同的 data_scope 主体
- 它们之间没有独立的用户决策点（用户不会只执行其中一个而跳过另一个）
- **注意**：FP合并是功能点层面的合并，不影响场景步骤数。合并后的FP在场景中仍可作为多个步骤引用

**拆分规则**（满足任一条件时拆分为多个FP）：
- 一个步骤包含"并且"/"然后"等连接词暗示多个独立操作
- 文档对该步骤有多组不同参数说明
- 该步骤有独立的成功/失败判定条件

每个 FP 记录：id, name, entry, priority, constraints, source_ids（["FN_NNN"]）。

**constraints 提取维度**：对每个FP扫描三类约束：
- 数值约束（范围/上限/下限，如"最多重试3次"、"响应时间<2s"）
- 格式约束（数据类型/正则/枚举值，如"密码长度8-20位"）
- 时间约束（超时/有效期/过期策略）

**参数覆盖自检**：遍历文档参数表/配置项，检查每个参数是否出现在某FP的constraints中。未覆盖的补入。

### 第五步：流程场景生成

将3c识别的主流程映射为 flow 场景。

**拆场景的唯一条件：两个场景的步骤序列描述不同的用户操作，而非同一操作的不同阶段/配置/参数。**

**禁止拆场景的情况（反模式）**：
- ❌ 完整链路的前半段和后半段 → 应合并（如"创建+配置"和"执行"应合并为"创建→配置→执行→停止"）
- ❌ 某个配置项的独立验证 → 应为已有场景的parameter分支（如"配置存储层"是"创建团队"的配置变体）
- ❌ 生命周期操作（初始化/启动/停止）单独成场景 → 应作为已有场景的步骤
- ❌ 已有场景的parameter分支已覆盖的操作 → 不再独立成场景

**映射规则**：

| 对象 | 产出 |
|------|------|
| 每条主流程 | 1个场景（含完整端到端steps + verify_points） |
| 未覆盖FP | 先判断该FP的性质：独立操作链路→补充场景；已有场景的后续步骤→合并到该场景；配置参数差异→作为parameter分支；生命周期操作→作为已有场景的步骤。禁止机械地为每个未覆盖FP创建独立场景 |
| 跨流程交互 | 独立场景（识别条件：两个流程共享 data_scope、或存在时序依赖、或一者的 verify_points 是另一者的前提条件） |

**steps 映射**：
- seq: 步骤序号
- action: 主流程步骤→用户操作语言
- fp_ref: 引用的FP
- data_scope: 从主流程继承
- check: 该步骤成功判定

**verify_points**: 主流程的端到端验证点（happy path预期结果）。

### 第六步：分支展开

对第五步的每个场景，从3c的信息展开6类分支：

| 分支类型 | 来源 | 展开方式 |
|---------|------|---------|
| parameter | variants + 参数配置差异 | 每个变体=1个分支。**去重**：已被第五步反模式排除的配置项不再展开 |
| boundary | 数值边界值（如最小值、最大值、空值） | 临界值=1个分支。枚举参数的非法值归入boundary而非constraint |
| exception | error_paths | 每个失败点=1个分支 |
| quality | 质量属性指标 | 每个指标=1个分支（如文档无显式指标可跳过） |
| constraint | FP的constraints中非枚举型约束 | 仅数值范围/超时等连续型约束→1个分支。枚举型约束已在boundary中覆盖，不重复展开 |
| cross | 跨FP/跨分支组合 | 满足交叉条件时=1个分支 |

每个分支记录：
- id: FS-{N}-{类型缩写}{n}
- description: 简述
- step_ref: 偏离的步骤序号（quality无此字段）
- trigger: 触发条件
- expected: 预期结果
- 类型特有字段：parameter有param/values，quality有risk_ref，constraint有constraint，cross有cross_refs/fp_refs

**cross 分支识别条件**（满足任一即生成）：

| 交叉模式 | 识别方法 |
|---------|---------|
| 参数×异常 | 同场景内：某 parameter 分支的参数取值影响某 exception 分支的触发条件或预期结果 |
| 约束传导 | 某步骤的 constraint 分支约束值传导到后续步骤，改变后续步骤的 boundary 触发值 |
| 跨FP数据依赖 | 场景内两个步骤的 fp_ref 不同但 data_scope 有交集（一个写、一个读同一实体） |

每条 cross 分支必须明确写出**组合触发条件**和**组合预期结果**，不能只复述单分支内容。

**cross 分支硬上限**：每个场景的 cross 分支数 ≤ 5。

超出上限时按以下优先级截断（优先保留高优先级）：

| 截断优先级 | 交叉模式 | 保留理由 |
|-----------|---------|---------|
| 1（必保） | 参数×异常 | 同时验证参数配置对异常行为的影响，双重验证价值高 |
| 2 | 约束传导 | 级联效应，一个约束触发后续连锁反应，发现隐蔽缺陷 |
| 3（先截） | 跨FP数据依赖 | 数据流一致性验证 |

**截断操作**：按优先级排序 → 保留前5个 → 剩余记录到 `truncated_cross` 字段（仅 id + description，不展开完整分支）。

### 第六步（续）：分支完整性追问

对每个场景执行递归追问，发现6类分支模板未覆盖的测试情况（不产生新场景，只补充分支）。

#### 追问执行

对每个场景依次执行：

**核心问题**：「这个场景的6类分支还有什么测试情况没有覆盖？」

**递归规则**：
1. 提出核心问题，等待模型自由思考
2. 模型回答的新测试情况：
   - 能归入6类 → 补充到对应分支
   - 无法归入 → 添加到 cross 分支
3. 对每个新发现继续追问：「这个新情况内部还有什么子情况？」
4. 终止条件：连续2轮无新发现，或已达最大5轮

#### 追问示例

```
场景：循环组件
追问1：「还有什么没覆盖？」→ 发现：嵌套循环、中断后状态、动态数据源
追问2：「嵌套循环还有什么子情况？」→ 发现：内层中断影响外层、变量污染
追问3：「还有什么？」→ 发现：break vs continue差异
追问4：「还有什么？」→ 无新发现 → 终止
```

#### 输出记录

在每个场景JSON中添加追问发现：

```json
{
  "branches": {
    "cross": [
      {"id": "FS-001-X05", "description": "嵌套循环内层中断影响外层", "trigger": "...", "expected": "..."}
    ]
  },
  "exploration_log": {
    "追问轮次": 3,
    "新发现数": 5,
    "终止原因": "连续2轮无新发现"
  }
}
```

### 第八步：自检

| 检查项 | 要求 |
|--------|------|
| FP粒度合规 | 每个FP满足三原则（单入口/单行为/可验证）？name包含"并且/然后"等连接词→应拆分？ |
| FP合并正确 | 被合并的FP之间确无独立用户决策点？ |
| FP参数覆盖 | 文档参数每个在某FP的constraints中？ |
| 场景FP覆盖 | 每个FP被至少一个场景的steps引用？ |
| 主流程覆盖 | 每条主流程有对应场景？ |
| **链路完整性** | **每个场景的steps构成完整端到端链路？没有场景是某个更长链路的前半段或后半段？（如"创建+配置"和"流式执行"应合并）** |
| **场景非子集** | **没有场景A的steps完全被场景B的steps或parameter分支包含？（如"配置存储层"被"创建团队"的parameter分支覆盖）** |
| **场景非包含** | **没有场景A的所有步骤出现在场景B中作为连续子序列？（如"Runner生命周期"是完整链路的子集）** |
| error_paths | 每个error_path有对应exception分支？ |
| variants | 每个variant有对应parameter分支？ |
| data_scope | 文档提到的数据类型在某步骤的data_scope中？ |
| data_scope交叉 | 共享 data_scope 的 FP 对都识别了 cross 分支？ |
| 跨场景依赖 | 有数据依赖关系的场景对都生成了跨流程交互场景？ |
| 追问执行 | 每个场景都执行了递归追问并记录exploration_log？ |
| cross分支上限 | 每个场景的cross分支数 ≤ 5？（超出时按截断优先级：参数×异常 > 约束传导 > 跨FP依赖） |
| 术语一致性 | 同一概念在FP/场景/分支中使用相同名称，不混用同义词 |
| 引用一致性 | 所有fp_ref/source_ids指向的ID均存在 |
| 文件完整性 | s1_index.json 的 scenario_index 条目数 = s1_scenarios/ 下的文件数？ |
| 禁止代码 | 全程未Read/Grep/Glob代码文件？ |
